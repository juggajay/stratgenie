## Framework Overview

### Purpose

This framework provides a reusable, structured approach to:
- **Plan content strategically** around topic clusters
- **Organize content calendars** by time and priority
- **Build topical authority** through pillar-spoke architecture
- **Map search intent** across content types
- **Prevent cannibalization** through clear content differentiation
- **Integrate SEO agents** into a coordinated workflow

### Core Principles

1. **Topic Clustering**: Organize content around main topics (pillars) with related subtopic supporting content (spokes)
2. **User-Centric Design**: Content answers specific user needs mapped to search intent
3. **Logical Progression**: Content builds topical authority progressively from broad to deep
4. **Differentiation**: Each piece of content serves a unique purpose with distinct targeting
5. **Scalability**: Framework works for small blogs (5-10 topics) or large sites (100+ topics)

### Target Users

- SEO strategists planning annual content roadmaps
- Content teams executing multi-week publishing schedules
- Authority sites building comprehensive topic clusters
- Site owners optimizing for topical relevance

## Topic Cluster Methodology

### What is a Topic Cluster?

A topic cluster is a content structure where:
- **Pillar Page** (Hub): Comprehensive, broad-topic guide covering all subtopics at high level
- **Spoke Content** (Cluster): Detailed articles targeting specific subtopics, linked back to pillar
- **Internal Linking**: Hub-spoke topology creates topical relevance signals for search engines

### Cluster Architecture Template

```
TOPIC CLUSTER: [Main Topic]
├─ PILLAR PAGE (Hub)
│  ├─ Title: [Topic] Complete Guide
│  ├─ Word Count: 2,500-5,000
│  ├─ Scope: Broad overview of all subtopics
│  └─ Internal Links: Link to all spokes
│
├─ SPOKE 1: [Subtopic A]
│  ├─ Title: How to [Subtopic A]
│  ├─ Word Count: 1,500-2,500
│  ├─ Scope: Deep dive into specific aspect
│  └─ Backlink: Link to pillar with contextual anchor
│
├─ SPOKE 2: [Subtopic B]
│  ├─ Title: [Subtopic B] Best Practices
│  ├─ Word Count: 1,500-2,500
│  ├─ Scope: Another specific aspect
│  └─ Backlink: Link to pillar with contextual anchor
│
├─ SPOKE 3: [Subtopic C]
│  ├─ Title: [Subtopic C] Step-by-Step Guide
│  ├─ Word Count: 1,500-2,500
│  ├─ Scope: Process/methodology focus
│  └─ Backlink: Link to pillar with contextual anchor
│
├─ SUPPORTING CONTENT
│  ├─ FAQ Page: Address common questions
│  ├─ Glossary: Define key terms
│  ├─ Comparison: [Topic] vs. alternatives
│  ├─ Tools/Resources: Curated list
│  └─ Case Study: Real-world example
│
└─ METADATA & STRUCTURE
   ├─ Cluster Keyword: [Primary topic phrase]
   ├─ Entity: [Brand/Topic entity name]
   ├─ Target Audience: [User persona]
```

### Content Calendar Best Practices

1. **Pillar-First Approach**: Always publish pillar pages before spokes (they provide context)
2. **Cluster Cohesion**: Complete clusters before starting new ones (builds topical authority faster)
3. **Buffer Time**: Schedule 20% of calendar as buffer for urgent content/trends
4. **Review Cycles**: Build in 1-week review periods between phases
5. **Cross-Team Coordination**: Plan handoffs between writers, editors, and optimizers
6. **Content Dependencies**: Map which pieces depend on others being published first

## Pillar Page Strategy

### What Makes a Great Pillar Page

A pillar page is the authoritative hub for a topic cluster. It should:
- **Comprehensively cover** the topic at a broad level
- **Address all major subtopics** that will have dedicated spoke articles
- **Be longer and deeper** than individual spoke articles
- **Include internal links** to all related spoke content
- **Serve as reference** for beginners discovering the topic
- **Rank for the primary keyword** and related variations

### Pillar Page Creation Checklist

- [ ] **Keyword Research**
  - [ ] Primary keyword identified and search volume verified
  - [ ] Related keyword variations identified
  - [ ] Search intent confirmed (informational/commercial/transactional)
  - [ ] Competitor analysis completed

- [ ] **Structure & Outline**
  - [ ] H1 includes primary keyword naturally
  - [ ] 4-6 main sections (H2s) covering major subtopics
  - [ ] 2-3 H3s per H2 for deeper organization
  - [ ] Table of contents included
  - [ ] Conclusion with clear next steps

- [ ] **Content Depth**
  - [ ] 2,500-5,000 words minimum
  - [ ] Each subtopic explained thoroughly but linked for deeper dives
  - [ ] Data, examples, and use cases included
  - [ ] Visual elements planned (diagrams, infographics)
  - [ ] FAQ section addressing top questions

- [ ] **E-E-A-T Signals**
  - [ ] Author credentials/bio included
  - [ ] Authoritative sources cited
  - [ ] Original research or data presented
  - [ ] Expert quotes or perspectives included
  - [ ] Trust signals (security, privacy, disclaimers)

- [ ] **Internal Linking Strategy**
  - [ ] Internal links planned to 5+ spoke articles
  - [ ] Contextual anchor text with natural keywords
  - [ ] Links placed naturally within sections (not forced)
  - [ ] "Read more about X" sections linking spokes
  - [ ] Related articles widget at end

- [ ] **SEO Technical**
  - [ ] H1-H6 hierarchy is logical and semantic
  - [ ] Primary keyword density 0.8-1.2%
  - [ ] Secondary keywords naturally distributed
  - [ ] Schema markup planned (Article, BreadcrumbList)
  - [ ] Meta description (150-160 chars) written

- [ ] **User Experience**
  - [ ] Introduction compelling and clear value statement
  - [ ] Scannable with short paragraphs and bullets
  - [ ] Clear sections with descriptive headings
  - [ ] Call-to-action to related spokes
  - [ ] Mobile-friendly format

## Supporting Content Hierarchy

### Content Type Hierarchy by Value

```
TIER 1: PILLAR PAGES
├─ Authority: Highest
├─ Traffic Potential: Highest
├─ Build Time: 4-8 weeks
└─ Purpose: Topic foundation

TIER 2: PRIMARY SPOKES (How-to, Deep Dives)
├─ Authority: High
├─ Traffic Potential: High
├─ Build Time: 2-4 weeks
└─ Purpose: Detailed subtopic coverage

TIER 3: SECONDARY SPOKES (Guides, Comparisons)
├─ Authority: Medium-High
├─ Traffic Potential: Medium-High
├─ Build Time: 1-3 weeks
└─ Purpose: Additional angles and perspectives

TIER 4: SUPPORTING CONTENT (FAQ, Resources, Glossary)
├─ Authority: Medium
├─ Traffic Potential: Medium (long-tail keywords)
├─ Build Time: 1-2 weeks
└─ Purpose: User engagement, featured snippets

TIER 5: CONTENT VARIATIONS (Infographics, Videos, Checklists)
├─ Authority: Medium
├─ Traffic Potential: Medium (referral, time-on-page)
├─ Build Time: 1-2 weeks
└─ Purpose: Multi-format consumption
```

### Supporting Content Type Definitions

#### 1. **Detailed Spokes** (Primary Supporting Content)

Purpose: Deep dive into specific subtopic with actionable detail

Structure:
- Title: Problem-focused or how-to format
- Length: 1,500-2,500 words
- Format: Tutorial, guide, best practices
- Links: Back to pillar + to related spokes
- Example: "How to [Specific Task]" or "[Subtopic] Best Practices"

When to Create:
- When pillar has high search volume for subtopic
- When user intent warrants detailed explanation
- When multiple variations of the subtopic exist

#### 2. **FAQ Pages**

Purpose: Address common user questions and capture featured snippet opportunities

Structure:
- 10-15 question-answer pairs
- Questions: Natural language (matching search queries)
- Answers: Concise but complete (200-400 words each)
- Format: Accordion or Q&A layout
- Schema: FAQ schema markup for rich results

When to Create:
- After pillar and primary spokes published
- When user questions patterns identified
- To address long-tail keyword variations

#### 3. **Glossaries & Terminology Pages**

Purpose: Define key terms and build entity relationships

Structure:
- 15-30 key terms
- Definitions: 50-150 words each
- Organization: Alphabetical or by category
- Links: Internal references between terms
- Schema: FAQPage or DefinitionSet schema

When to Create:
- For technical or jargon-heavy topics
- To build semantic relevance signals
- To capture "what is X" search queries

#### 4. **Comparison & Versus Content**

Purpose: Help users choose between options

Structure:
- 1,500-2,500 words
- Format: Feature comparison table + narrative analysis
- Comparisons: 3-5 options (pillar option vs. alternatives)
- Schema: ComparisonTable schema
- Angle: Bias toward your pillar topic

When to Create:
- When users compare your approach to competitors
- To capture "versus" and "alternatives" searches
- To guide users toward deeper engagement

#### 5. **Case Studies & Real-World Examples**

Purpose: Demonstrate practical application and build trust

Structure:
- 1,500-3,000 words
- Format: Problem-Solution-Results narrative
- Elements: Customer background, challenge, approach, results, metrics
- Links: Reference to relevant spokes
- Length: Detailed but scannable

When to Create:
- For YMYL or high-consideration topics
- To build trust and social proof
- To demonstrate topical expertise

#### 6. **Tool Lists & Resource Pages**

Purpose: Curate and review relevant tools/resources

Structure:
- 1,500-2,500 words
- Format: List with descriptions and comparisons
- Items: 5-15 tools/resources with pros/cons
- Organization: Category-based
- Links: External + internal to related spokes

When to Create:
- To provide ongoing value to audience
- To rank for "best tools for X" searches
- To increase time-on-site and engagement

#### 7. **Checklists & Templates**

Purpose: Provide downloadable, actionable resources

Structure:
- Organized checklist format (2-5 sections)
- Downloadable PDF or interactive format
- 50-150 items depending on complexity
- Tied to specific spoke article
- Links: Back to related detailed articles

When to Create:
- For how-to and process-oriented content
- To increase email list signups
- To improve user engagement metrics

#### 8. **Infographics & Visual Content**

Purpose: Communicate complex ideas visually

Structure:
- One main concept/process visualized
- Complementary to pillar or spoke
- Sized for social sharing (1200+ width)
- Alt text with keywords
- Embedded on pillar + dedicated post

When to Create:
- For process-heavy topics
- To increase social shares
- To improve time-on-page metrics

## Search Intent Mapping

### Search Intent Categories

Search intent defines the "why" behind a search query. Content must align with user intent to rank and satisfy users.

### Intent Type Framework

#### 1. **Informational Intent**
*User wants to learn or understand something*

**Characteristics**:
- Queries: "What is X", "How does X work", "Best practices for X"
- User stage: Early awareness
- Content type: Guides, overviews, explanations, how-tos

**Content Approach**:
- Comprehensive, beginner-friendly
- Educational tone
- Many internal links to related topics
- FAQ sections
- Visual explanations

**Keywords**: "[Keyword] meaning", "[Keyword] explained", "[Keyword] guide"

#### 2. **Navigational Intent**
*User wants to find a specific site or resource*

**Characteristics**:
- Queries: Brand names, branded + term, site: searches
- User stage: Know what they want, finding the resource
- Content type: Landing pages, branded content

**Content Approach**:
- Clear brand/product positioning
- Easy navigation to key pages
- Authority establishment (why use us)
- Trust signals prominent
- Search form integration

**Keywords**: "[Brand] + [term]", "[Product name]", "best [category]"

#### 3. **Transactional Intent**
*User wants to complete an action (buy, signup, download)*

**Characteristics**:
- Queries: "Buy X", "Sign up for X", "Download X"
- User stage: Ready to convert
- Content type: Product pages, pricing pages, signup pages

**Content Approach**:
- Clear value proposition at top
- Compare options/pricing
- Clear CTA buttons
- Trust signals (reviews, guarantees)
- Friction reduction
- Minimal navigation away from goal

**Keywords**: "Buy [keyword]", "[keyword] pricing", "[keyword] free trial"

#### 4. **Commercial Investigation**
*User researching to make a purchase decision*

**Characteristics**:
- Queries: "Best X for Y", "X vs Y", "Top X reviews"
- User stage: Comparison/evaluation
- Content type: Comparisons, reviews, alternatives

**Content Approach**:
- Comparative analysis
- Pro/con breakdowns
- Feature comparisons
- Social proof (reviews, testimonials)
- Recommendation rationale
- Links to product pages

**Keywords**: "Best [keyword] for [use case]", "[keyword] comparison"

## Final Recommendations

### Framework Success Factors

1. **Commit to Topics**: Complete clusters fully before starting new ones
2. **Measure Continuously**: Track metrics monthly, assess quarterly
3. **Coordinate Across Agents**: Use handoff protocol for consistent execution
4. **Build Authority Progressively**: Depth > speed in topical authority
5. **Update Regularly**: Refresh content annually and as landscape changes
6. **Monitor Competition**: Stay aware of competitor content moves
7. **User Focus First**: Optimize for user needs before search engine signals
8. **Quality Over Quantity**: One great pillar beats five mediocre articles

### Common Pitfalls to Avoid

- Publishing spokes before pillar (lose context and internal linking benefit)
- Starting new clusters before finishing existing ones (dilutes authority signals)
- Neglecting internal linking (miss topical relevance signals)
- Over-optimizing keywords (reduces readability, damages rankings)
- Ignoring search intent (high bounce rates, poor engagement)
- Forgetting to update older content (stale information damages authority)
- Publishing without E-E-A-T signals (particularly important for YMYL topics)
- Abandoning clusters too early (topical authority takes 6-12 months minimum)
